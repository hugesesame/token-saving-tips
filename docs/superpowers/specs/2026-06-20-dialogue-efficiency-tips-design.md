# 対話の伝え方Tips統合 — 設計スペック

## 背景・目的

ユーザーがClaude.aiチャットでのやり取りを振り返り、「往復(ターン数)を減らすための伝え方・指示の出し方」に関する気づきを共有してくれた。これを既存のtoken-saving-tipsサイト（Claude Codeでのトークン節約Tips集）に統合し、内容を深掘りする。

既存サイトは「Claude Codeの操作（コマンド・モデル選択・スコープ指定など）で節約する」観点のTipsのみで構成されている。今回追加するのは「指示の出し方・対話の仕方で往復（≒トークン）を減らす」という別観点であり、既存観点と並走させる形で各レベル（初級/中級/上級）に組み込む。

## スコープ

- 対象ファイル: `index.html`, `css/style.css`（JSの変更は不要）
- 各レベル（初級/中級/上級）のタイル内を、サブ見出しで「操作で節約」「伝え方で節約」の2グループに分割する
- 既存カードはすべて「操作で節約」グループに分類し直す（内容・コピーは変更しない）
- 新規カードを「伝え方で節約」グループとして各レベルに追加する
- カード番号はグループをまたいでタイル単位の連番を維持する（操作カードの直後から続き番号で新規カードを採番）

### スコープ外（やらないこと）

- 「claude.aiチャットからClaude Codeへ移行するとファイル再アップロードが不要になる」という話は、本サイトの読者がすでにClaude Codeを使っている前提のため、独立カードとしては追加しない
- 既存の上級カード「プランモードを積極的に使う」（操作系・上級③）の内容は変更しない。新規の対話系カードからは一文で相互参照するに留め、内容を重複させない
- レベル分けの基準（難易度ベース）やjump-nav、IntersectionObserverのロジックは変更しない

## 構造設計

### HTML構造

各 `<section class="tile">` 内、`.tile__header` の直後を以下のように変更する。

```html
<div class="tile__header">...</div>

<div class="tip-group">
  <p class="tip-group__label">操作で節約</p>
  <div class="tip-grid">
    <!-- 既存カードをそのまま移動（連番は変更しない） -->
  </div>
</div>

<div class="tip-group">
  <p class="tip-group__label">伝え方で節約</p>
  <div class="tip-grid">
    <!-- 新規カード（既存カードの続き番号） -->
  </div>
</div>
```

### CSSスタイル（新規追加）

```css
.tip-group {
  margin-top: var(--space-2xl);
}

.tip-group:first-child {
  margin-top: 0;
}

.tip-group__label {
  font-size: 13px;
  font-weight: 600;
  letter-spacing: 0;
  color: var(--color-ink-muted-48);
  padding-bottom: var(--space-xs);
  margin-bottom: var(--space-lg);
  border-bottom: 1px solid var(--color-hairline);
}

.tile--dark .tip-group__label {
  color: var(--color-body-muted);
  border-bottom-color: rgba(255, 255, 255, 0.12);
}
```

既存の `.tip-grid` 自体のスタイル（グリッド・gap・レスポンシブ）は変更しない。`.tip-group` でラップするだけなので、カードのレイアウト・折り返し挙動には影響しない。

## コンテンツ設計（新規カード）

各カードは既存カードと同じマークアップパターン（`tip-card` / `tip-card__number` / `tip-card__title` / `tip-card__body`、必要に応じて `example-label` + `code-block` + `example-note`）を踏襲する。

### 初級（既存1〜5の続き）

**⑥ 関連する質問はまとめて送る**

> 「バージョン確認」「ターミナルの開き方」「ショートカット一覧」のように関連する質問を1つずつ送ると、その都度ターンを消費します。最初に必要な情報をまとめて聞くだけで、やり取りの回数を大きく減らせます。

- 改善前:
  ```
  バージョンの確認方法を教えて

  （回答後）ターミナルの開き方を教えて

  （回答後）よく使うショートカットも教えて
  ```
- 改善後:
  ```
  バージョンの確認方法・ターミナルの開き方・
  よく使うショートカットを、まとめて教えて
  ```
- note: 関連する質問は最初から並べて聞くことで、回答も1往復で済みます。

**⑦ 環境情報は最初の一言に添える**

> OSやバージョンなど、回答が分岐しうる前提条件は質問と同時に伝えましょう。後から「今Macを使っているので」と言い直す分、往復が増えてしまいます。

- 改善前:
  ```
  ファイル検索のショートカットを教えて

  （回答後）今Macを使っているので、Mac版で教えて
  ```
- 改善後:
  ```
  （Mac環境です）ファイル検索のショートカットを教えて
  ```
- note: 一言加えるだけで、聞き直しの往復が丸ごと不要になります。さらに環境情報自体を毎回書きたくない場合は、中級「CLAUDE.mdに環境情報を常設する」が役立ちます。

### 中級（既存1〜7の続き）

**⑧ CLAUDE.mdに環境情報を常設する**

> CLAUDE.mdはセッションのたびに自動で読み込まれるため、OSやツールなどの前提条件を一度書いておけば、以後は毎回伝え直す必要がなくなります。

- 記述例（code-block）:
  ```
  ## 環境情報

  - OS: macOS (M5 MacBook Air, 24GB/1TB)
  - ターミナル: zsh
  - パッケージ管理: Homebrew, npm
  - エディタ: VSCode (`code` コマンド有効)

  ## 作業時の前提

  - Mac環境を前提に手順・ショートカット・パスを案内すること
  - パスは `~/` 形式で統一する
  - ショートカットはMacキー表記（Cmd, Option, Ctrl）で統一すること
  ```
- note: 「環境情報（事実）」と「作業時の前提（振る舞いの指示）」を分けて書いておくと、環境が変わったときにどちらを直せばいいか分かりやすくなります。

**⑨ 「条件付き実行」を指示に含める**

> 「まず調査→確認→実行」を毎回律儀に区切ると、その分だけ往復が増えます。「迷う場合だけ確認して」と判断基準を先に渡しておくと、調査から実行までを一括で任せられます。

- 改善前:
  ```
  「居酒屋の星」の下にイラストを移動して
  ```
- 改善後:
  ```
  「居酒屋の星」の下にイラストを移動して。
  該当箇所が複数あって判断に迷う場合だけ確認して、
  それ以外はそのまま反映してOK
  ```
- note: 「迷わない場合は聞かない」という基準を渡すことで、調査・提案・修正を1ターンにまとめられます。

**⑩ 複数の変更を1メッセージにまとめてバッチ化する**

> 「位置移動→フォントサイズ→アニメーション」のような連続した修正を1個ずつ送ると、その都度「やりました、次は？」というやり取りが挟まります。最初に全部リストアップしておきましょう。

- 例（code-block）:
  ```
  以下を順に直してください。
  途中で質問があれば全部まとめて最後に聞いて:

  1. イラストを〜の下に移動
  2. 「使用上の注意」のフォントサイズを上げる
  3. ホバー時のグロー効果を追加
  ```
- note: 慎重に進めたい場合でも、「確認は最後にまとめて」と伝えるだけで、確認のたびに発生する往復を1回に集約できます。

### 上級（既存1〜9の続き）

**⑪ diffプレビューを指定して確認往復を圧縮する**

> 実行前に必ず内容を見たい場合は、「実行して」ではなく「diffだけ出して」と指定しましょう。調査・修正・diff提示までを1ターンでまとめて行えます。

- 例（code-block）:
  ```
  移動して、変更箇所のdiffだけ見せて。
  問題なければ次のメッセージで「OK」と送るので、
  その時は確認なしでそのまま確定して
  ```
- note: 「diff提示」と「OK」の2往復だけで完結し、調査だけのための往復が消えます。ただし構造を大きく変える変更や元に戻しづらい操作は、念のため都度確認したほうが安全です。

**⑫ CLAUDE.mdに「標準確認ルール」を書いておく**

> 「迷ったら聞いて、それ以外はそのまま進めてOK」という基準を指示のたびに書くのではなく、CLAUDE.mdに常設しておくと、確認の粒度を毎回指定する手間がなくなります。プランモード（操作で節約・上級③）と組み合わせると、計画フェーズと実行フェーズの分離でも同様に往復を減らせます。

- 例（code-block）:
  ```
  ## 確認なしで進めてよい操作

  - 確認が必要な変更（削除・構造変更・公開設定の変更）以外は、
    聞かずにそのまま進めてOK
  - 迷うケースがあった場合のみ、まとめて質問してから進める
  ```
- note: ルールをCLAUDE.mdに一度書いておけば、以後のすべてのセッションで「確認の基準」をその都度伝える必要がなくなります。

## 影響範囲・互換性

- jump-nav / sub-nav のリンク先（`#beginner` `#intermediate` `#advanced`）は変更なし。IntersectionObserverの対象（`main section[id]`）も変更なし。
- 既存カードの文言・順序・スタイルクラスは変更しない（`.tip-group` でラップするのみ）。
- モバイル幅でのグリッド折り返し（`minmax(0, 1fr)` 対応済み）はそのまま適用される。新規カードも長い行を含む場合は `.code-block` の `overflow-x: auto` で横スクロールに収まる想定。

## 検証方法

- Playwright（システムChrome）で375px/414px/1280px幅のスクリーンショットを取得し、以下を確認する:
  - 横スクロール（overflow）が発生していないこと
  - `.tip-group__label` の見出しが各レベル・各グループで正しく表示されていること（特にdarkタイルでの配色）
  - 新規カードの折り返し・コードブロックの読みやすさ（先ほど修正したターミナルグリーン色が引き続き適用されていること）
